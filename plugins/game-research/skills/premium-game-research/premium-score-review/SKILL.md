# 买断制游戏评分审阅技能

对一份已完成的买断制游戏属性维度表评分报告进行独立审阅，发现事实性错误、评分逻辑矛盾和信源盲区。

审阅结果在对话中逐条呈现，由用户决定是否修改原报告。

## 输入

用户提供一份已完成的买断制游戏属性维度表报告（.md 文件路径）。

## 工作流程

### 第一步：加载上下文

读取以下文件：

1. 待审阅的报告文件
2. `premium-game-research/references/framework-definition-premium.md` — 维度定义与评分锚点
3. `premium-game-research/references/reference-scores-premium.md` — 参照游戏评分数据

### 第二步：Reference 数据同步检查

在开始审阅之前，先确认评分工具链本身是否存在过时问题：

- 报告中标注的框架版本号是否与当前 `framework-definition-premium.md` 的版本号严格一致（如报告标注 v1.0-premium，框架当前也是 v1.0-premium）。版本号不一致则整份报告的评分标准可能已过时，优先报告此问题。
- 报告中引用的参照游戏评分是否与当前 `reference-scores-premium.md` 一致
- 如果参照游戏中有已知的重大更新（如大型DLC发售、导演剪辑版、次世代升级等），检查 `reference-scores-premium.md` 中对应评分是否仍然准确

如果发现不一致，在审阅结果的开头优先报告，因为这会影响所有横向对比的可靠性。

### 第三步：事实断言提取

从报告的每条评分理由中，提取所有被当作事实使用的断言，列成清单。

重点关注以下类型：

| 类型 | 示例 |
|------|------|
| 系统存在性 | "游戏有同伴系统" "支持外观自定义" "存在NG+模式" |
| 角色设定 | "主角是可自建角色" "V有语音" "玩家选择影响结局" |
| 世界结构 | "无缝开放世界" "区域间有过场加载" "采用Hub-Spoke结构" |
| 核心循环 | "战斗占主要比重" "有装备收集-定制系统" "有建造系统" |
| 产品结构 | "有DLC扩展" "通关后开放新区域" "多结局设计" |

### 第四步：事实核查

对第三步提取的每条断言，使用 WebSearch 进行独立验证。

核查原则：
- 每条断言至少搜索一次，使用与报告中不同的关键词
- 优先查找一手来源（官方Wiki、开发商公告、游戏内系统说明）
- 搜索结果中找不到具体描述的系统/机制，视为"无法验证"而非默认存在
- 注意区分目标游戏与同系列其他作品的信息

对每条断言标注验证状态：
- ✅ **已验证** — 至少两个独立信源确认
- ⚠️ **单源** — 仅一个信源提及
- ❌ **已否定** — 搜索结果与报告中的断言矛盾
- ❓ **无法验证** — 信息不足

### 第五步：评分一致性检查

逐维度检查以下两类一致性问题：

**A. 评分-理由一致性**

评分数值是否与其理由匹配。

例：理由写"玩家在叙事层面没有决策权，所有玩家经历相同的主线"，但叙事分支密度打了5分（应该更低）。

**B. 跨维度张力识别**

识别不同维度之间看似矛盾的评分组合，提请用户关注。这些张力点往往对应有趣的设计决策，不一定是错误。

值得关注的模式举例：
- 叙事控制权高 + 叙事分支密度低 → 世界规则允许涌现行为，但叙事走向本身不分支（如塞尔达、艾尔登法环）
- NPC依恋密度高 + 自我代入深度低 → 玩家对特定角色有强烈情感，但不将自己投射为主角（如亚瑟·摩根）
- N1a-P 叙事节奏结构高分（叙事稀疏）+ 叙事推进循环权重高 → 叙事总量不大但玩家在叙事中的时间占比被高估，需确认
- 多结局/NG+权重高 + 单次通关权重也高 → 需确认游戏是否同时鼓励多周目和提供充实的单次体验

识别到张力点后，呈现给用户判断即可。如果理由中已经自洽地解释了这种张力，标注为"已解释"并跳过。

### 第六步：框架校准检查

将每个连续谱维度的评分对照框架定义中的低端/高端锚点描述，确认分数是否落在合理区间。

检查方式：
- 读取框架中该维度的1分描述和10分描述
- 判断报告中的理由描述更接近哪一端
- 如果理由描述接近低端但分数偏高（或反过来），标记为校准偏差

特别关注点：
- **N1a-P 叙事节奏结构**：1=全程叙事密度均匀，10=大量非叙事体验+叙事集中在少数节点。这是买断制特有的连续谱维度，容易与 GaaS 的 N1a 内容发布节奏混淆。
- **P1 叙事分支密度**：严格区分"选择有反馈"和"选择改变走向"。许多游戏的对话选择只改变局部反应，不改变整体叙事结构。

### 第七步：信源覆盖评估

检查报告的"评分说明与可信度"部分：
- 是否有维度的评分理由完全没有信源支撑（仅基于推断）
- 是否有维度过度依赖单一类型信源（如全部来自玩家社区讨论）
- 已知盲区是否被充分标注

### 第八步：呈现审阅结果

在对话中按以下结构呈现发现：

**严重问题**（事实性错误或明显的评分-理由脱节）：
- 每条列出：维度 → 问题描述 → 验证依据 → 建议修正

**设计张力点**（跨维度的有趣张力，提请用户关注）：
- 每条列出：涉及维度 → 张力描述 → 当前报告是否已解释

**需要注意**（信源不足或校准偏差，不一定是错误）：
- 每条列出：维度 → 疑点 → 建议核查方向

**确认无误**：
- 简要说明哪些维度通过了审阅

最后，给出审阅的整体结论：这份报告的可信度如何，哪些维度的评分可以直接使用，哪些建议修正后再使用。

## 注意事项

- 审阅的目标是提高报告准确性，不是推翻评分。如果评分合理但理由表述不够清晰，指出表述问题而非质疑分数。
- 不要因为某个评分"感觉不对"就标记为问题——只有在事实验证或框架校准中发现具体依据时才标记。
- 跨维度张力不等于错误。识别后交由用户判断，不要替用户下结论。
- 当核查搜索结果与报告内容矛盾时，同时呈现报告中的原始断言和核查搜索结果，让用户看到双方证据后自行裁决。不要单方面裁定哪个正确。
- 买断制游戏没有持续运营层，审阅重点应放在体验气质层（第一层）和产品结构层（第三层）的事实准确性上。
