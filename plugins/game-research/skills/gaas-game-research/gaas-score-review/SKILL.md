# GaaS 游戏评分审阅技能

对一份已完成的游戏属性维度表评分报告进行独立审阅，发现事实性错误、评分逻辑矛盾和信源盲区。

审阅结果在对话中逐条呈现，由用户决定是否修改原报告。

## 输入

用户提供一份已完成的游戏属性维度表报告（.md 文件路径）。

## 工作流程

### 第一步：加载上下文

读取以下文件：

1. 待审阅的报告文件
2. `gaas-game-research/references/framework-definition.md` — 维度定义与评分锚点
3. `gaas-game-research/references/reference-scores.md` — 参照游戏评分数据

### 第二步：Reference 数据同步检查

在开始审阅之前，先确认评分工具链本身是否存在过时问题：

- 报告中标注的框架版本号是否与当前 `framework-definition.md` 的版本号严格一致（如报告标注 v1.1，框架当前也是 v1.1）。版本号不一致则整份报告的评分标准可能已过时，优先报告此问题。
- 报告中引用的参照游戏评分是否与当前 `reference-scores.md` 一致
- 如果参照游戏中有已知的重大运营变化（如停服、大改版），检查 `reference-scores.md` 中对应评分是否仍然准确

如果发现不一致，在审阅结果的开头优先报告，因为这会影响所有横向对比的可靠性。

### 第三步：事实断言提取

从报告的每条评分理由中，提取所有被当作事实使用的断言，列成清单。

重点关注以下类型：

| 类型 | 示例 |
|------|------|
| 系统存在性 | "游戏有公会系统" "支持外观自定义" |
| 角色设定 | "主角是可自建角色" "协同者有20级好感度" |
| 机制数值 | "保底80抽" "体力上限240" |
| 内容结构 | "主线分章节推进" "限时活动有复刻" |
| 商业模式 | "角色Gacha是核心变现" "无订阅制" |
| 生命周期数据 | "该版本上线后Steam峰值XX万" "长草期从第3周开始" "内容消耗周期约2周" |

### 第四步：事实核查

对第三步提取的每条断言，使用 WebSearch 进行独立验证。

核查原则：
- 每条断言至少搜索一次，使用与报告中不同的关键词
- 优先查找一手来源（官方攻略站、Wiki、开发者公告）
- 搜索结果中找不到具体描述的系统/机制，视为"无法验证"而非默认存在
- 注意区分目标游戏与同系列其他作品的信息

对每条断言标注验证状态：
- ✅ **已验证** — 至少两个独立信源确认
- ⚠️ **单源** — 仅一个信源提及
- ❌ **已否定** — 搜索结果与报告中的断言矛盾
- ❓ **无法验证** — 信息不足

### 第五步：评分一致性检查

逐维度检查以下两类一致性问题：

**A. 评分-理由一致性**

评分数值是否与其理由匹配。

例：理由写"玩家在叙事层面没有决策权"，但叙事控制权打了5分（应该更低）。

**B. 跨维度张力识别**

识别不同维度之间看似矛盾的评分组合，提请用户关注。这些张力点往往对应有趣的设计决策，不一定是错误。

值得关注的模式举例：
- NPC依恋密度高 + 社交结构"独立"权重高 → 角色依恋不通过社交系统实现，这是一种设计选择
- 叙事控制权低 + 玩家能动性高 → 叙事由设计者控制但非叙事层面有高能动性
- 版本叙事权重高 + 内容发布节奏标"活动驱动" → 需确认哪个是主导

识别到张力点后，呈现给用户判断即可。如果理由中已经自洽地解释了这种张力，标注为"已解释"并跳过。

### 第六步：框架校准检查

将每个连续谱维度的评分对照框架定义中的低端/高端锚点描述，确认分数是否落在合理区间。

检查方式：
- 读取框架中该维度的1分描述和10分描述
- 判断报告中的理由描述更接近哪一端
- 如果理由描述接近低端但分数偏高（或反过来），标记为校准偏差

### 第七步：信源覆盖评估

检查报告的"评分说明与可信度"部分：
- 是否有维度的评分理由完全没有信源支撑（仅基于推断）
- 是否有维度过度依赖单一类型信源（如全部来自玩家社区讨论）
- 已知盲区是否被充分标注

### 第八步：呈现审阅结果

在对话中按以下结构呈现发现：

**严重问题**（事实性错误或明显的评分-理由脱节）：
- 每条列出：维度 → 问题描述 → 验证依据 → 建议修正

**设计张力点**（跨维度的有趣张力，提请用户关注）：
- 每条列出：涉及维度 → 张力描述 → 当前报告是否已解释

**需要注意**（信源不足或校准偏差，不一定是错误）：
- 每条列出：维度 → 疑点 → 建议核查方向

**确认无误**：
- 简要说明哪些维度通过了审阅

最后，给出审阅的整体结论：这份报告的可信度如何，哪些维度的评分可以直接使用，哪些建议修正后再使用。

## 注意事项

- 审阅的目标是提高报告准确性，不是推翻评分。如果评分合理但理由表述不够清晰，指出表述问题而非质疑分数。
- 不要因为某个评分"感觉不对"就标记为问题——只有在事实验证或框架校准中发现具体依据时才标记。
- 跨维度张力不等于错误。识别后交由用户判断，不要替用户下结论。
- 当核查搜索结果与报告内容矛盾时，同时呈现报告中的原始断言和核查搜索结果，让用户看到双方证据后自行裁决。不要单方面裁定哪个正确。
